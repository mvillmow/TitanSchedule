Tournament Sorting Network

A Monolithic Graph Architecture for Visualizing Volleyball Schedules

The User Experience Gap

Current scheduling systems (e.g., SportsEngine AES) scatter tournament data across multiple paginated views. A user determining their next game location often has to traverse 3-4 distinct pages, mentally tracking pool results and bracket cross-references.

Objective: Collapse this multi-step navigation into a single, monolithic graph where the left side represents the start of the day and the right side represents the final standing.

Figure 1: Efficiency Comparison (Estimated)

The Sorting Network Concept

Instead of static tables, we treat the tournament as a Sorting Network. Teams are inputs that pass through "Comparator Nodes" (Matches). Winners sort 'up' to higher brackets, losers sort 'down'. The state is versioned after every match result.

Engineering Principles

To ensure maintainability and robustness, the system adheres to strict software engineering principles. Simplicity (KISS) and modularity are prioritized to handle the complexity of tournament logic.

Automation Pipeline

The system runs "Headless". There is no traditional backend server. GitHub Actions serves as the cron-job runner, fetching data, processing the graph, and deploying static assets.

SOURCE SportsEngine AES (HTML/API)
GITHUB ACTION (CRON) Scraper & Graph Builder (Python)
ARTIFACT GENERATION JSON Graph & PNG/PDF Export
DEPLOYMENT GitHub Pages (SPA Client)

Path Prediction Complexity

A key feature is "Future Pathing". When a user selects a team, the system calculates all valid remaining nodes. This heatmap visualizes the density of potential final placements based on current seed positions.
